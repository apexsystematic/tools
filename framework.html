Phase 1 — Discovery & qualification

Understand the client's business ops before touching any tool. The goal is uncovering real pain, not just collecting a wishlist.

Steps
✓
Run intake call (DRI score context) 30 min

Use the Apex Systematic DRI framework as your anchor — this isn't a generic discovery call, it's a structured ops audit. Ask about the three DRI dimensions upfront.

  • What does a typical week look like across revenue, delivery, and ops admin?
  • Where do things fall through the cracks most often?
  • What's the most repetitive task that eats time but adds no thinking?
  • What tools are currently in use — and which ones actually get used?
Deliverable
  • Intake call notes with pain signals ranked
  • Rough DRI score estimate
✓
Identify the 3 highest-friction workflows Analysis

Don't automate everything at once. Lock in on three workflows that are: high-frequency, manual-heavy, and clearly defined. Vague workflows get automated wrong.

  • How many times per week does this happen?
  • Who touches it and how many handoffs are involved?
  • Is there a clear trigger and a clear end state?
Watch for scope creep bait — clients love to say "and also..." during discovery. Log it, don't commit to it.
✓
Qualify the engagement Go/no-go

Not every client is worth taking. Run this quick checklist before quoting anything.

  • Can they clearly describe the problem they want solved?
  • Do they have (or can they get) API access to their current tools?
  • Are they willing to change how they work, or do they want automation on top of chaos?
  • Is there a budget signal — or are they expecting magic for free?
Red flag: client wants to automate something they haven't done manually yet. You can't automate a process that doesn't exist.
✓
Define success criteria upfront Alignment

Get explicit agreement on what "done" looks like before the engagement starts. This protects you and anchors the client.

  • What metric improves and by how much?
  • How many hours per week should this save?
  • What's the one workflow that, if automated, would make this worth the investment?
Deliverable
  • 1-page brief: problem, workflows, success metric, agreed scope
🗺️

Phase 2 — Process mapping

Before writing a single automation, document exactly how the work flows today. Most automation failures trace back to skipping this step.

Steps
✓
Shadow or walk through the current process Async/sync

Have the client walk you through the process live — screen share, Loom, or a detailed written walkthrough. You're looking for every step, every app, every decision point, and every place they go off-script.

  • Ask them to do it as if you weren't watching — don't let them present their ideal version
  • Note every app they switch to (tab switching = integration opportunity)
  • Flag every decision that requires judgment — that's where AI can help
  • Capture edge cases they mention in passing
✓
Build the current-state process map Documentation

Document every step, trigger, tool, and decision in the existing workflow. Simple is fine — a numbered list with the tool per step works. No need for fancy flowchart software.

  • Trigger: what starts this process?
  • Steps: what happens in sequence?
  • Tools: what app handles each step?
  • Decisions: what varies and why?
  • End state: what does done look like?
Deliverable
  • Current-state process map (per workflow)
  • Tool inventory with integration capability notes
✓
Identify integration touchpoints Technical

Map every tool in the current process against your stack. Determine what's native, what needs Make.com, and what needs a custom webhook or API call.

Integration typeWhen to use
Native Make.com moduleTool has a direct Make connector — easiest path
HTTP / webhook moduleTool has an API but no Make module
Email parsingTool has no API — trigger off email output
Zapier bridgeTool only works in Zapier ecosystem
✓
Design the future-state process Design

Now rebuild the process with automation applied. For each manual step, decide: eliminate, automate, or keep human. Not everything should be automated — some steps need judgment.

  • Eliminate: steps that only existed to compensate for a missing tool
  • Automate: high-frequency, rule-based, low-stakes steps
  • AI-augment: steps requiring language, summarisation, or classification
  • Keep human: approvals, client-facing decisions, creative judgment
Deliverable
  • Future-state map with automation layer annotated
  • Client sign-off before build begins
⚙️

Phase 3 — Automation design

Translate the future-state map into a concrete automation architecture. Decide tools, triggers, data flow, and AI prompts before building anything.

Steps
✓
Select the trigger mechanism Architecture

Every automation starts with a trigger. Getting this right is critical — a bad trigger means the automation fires at the wrong time or not at all.

Trigger typeBest for
Tally form submitClient intake, onboarding, requests
Calendly event createdBooking confirmations, prep sequences
Gmail label / new emailInbound lead or support routing
Airtable record updateStatus changes, stage progressions
Webhook (custom)Any app that can POST a payload
Schedule (cron)Recurring reports, digests, cleanups
✓
Design the data model Data

Map every piece of data that flows through the automation. What comes in via the trigger, what gets transformed, what gets created, and where it lives at the end.

  • What fields arrive in the trigger payload?
  • What data needs to be looked up from another system?
  • What does Claude need to receive in its prompt, and what does it return?
  • Where does the final output live — Airtable, Google Drive, Gmail, Brevo?
Name your variables clearly in Make.com. Bundle.field notation gets messy fast — use Set Variable modules to alias long paths before passing data downstream.
✓
Write and test AI prompts AI layer

If Claude is in the workflow, design and test the prompt before wiring it into Make. Bad prompts cause silent failures — the automation runs, but the output is garbage.

  • Prompt structure: role, context, task, output format, constraints
  • Use explicit output format instructions (JSON, numbered list, specific fields)
  • Test with edge-case inputs, not just the happy path
  • Parse via content[0].text in the Make HTTP module response
Anthropic API Make HTTP module JSON parse
Deliverable
  • Prompt document with tested variations
  • Sample output for each prompt
✓
Wire the scenario blueprint in Make Make.com

Build the module-by-module architecture in Make.com before connecting live data. Use dummy data and test webhooks to validate the flow end-to-end in the design phase.

  • Start from trigger, work right — never build backwards
  • Add error handlers on every critical module (not just the last one)
  • Use filters to gate paths early — cheaper than routing errors downstream
  • Label every module clearly — you'll be back in 3 months

Phase 4 — Build & test

Execute the design. Build in layers — get the skeleton working before adding AI, error handling, or edge-case logic.

Steps
✓
Build the happy-path scenario first Core build

Build only the main flow with real data. Get a clean test run from trigger to end state before layering on anything else. This is your baseline — everything else is an edge case.

  • Use real (but non-critical) data for tests — fake data hides real parsing issues
  • Confirm every module receives what it expects before moving to the next
  • Verify outputs land in the correct destination fields
Do not add error handling, filters, or edge cases until the happy path runs clean. Adding complexity to a broken base makes debugging exponentially harder.
✓
Add AI layer and validate output AI integration

Drop Claude into the flow via the Make HTTP module. Feed it real data from the upstream modules and validate that the response is parsed correctly and passed downstream.

  • POST to https://api.anthropic.com/v1/messages
  • Set anthropic-version header and x-api-key
  • Parse response body → content[0].text
  • If Claude returns JSON, run it through Make's JSON parse module before mapping fields
  • Test with 5+ real inputs — not just one
If Claude's output format varies across runs, your prompt isn't specific enough. Add stricter format constraints and retest before wiring downstream modules.
✓
Build error handling and fallbacks Resilience

Every automation will fail eventually. Build for it before handoff — not after the client calls you at 9pm because something broke.

  • Add error handlers on: API calls, data lookups, file creation, email sends
  • Route failures to a Slack alert or email notification (to you, not the client)
  • Use Make's "Break" error directive for anything that requires a retry
  • Document what manual recovery looks like for each failure point
✓
Run full QA with edge cases Testing

Test every variation of input the automation will realistically encounter. Don't just test what should work — test what shouldn't and confirm the automation handles it gracefully.

  • Missing required fields in trigger payload
  • Duplicate submissions (same data triggered twice)
  • Long text inputs that might overflow field limits
  • Special characters or formatting in text fields
  • Upstream API being slow or unavailable
Deliverable
  • QA log with all test cases and results
  • Sign-off checklist before handoff
🚀

Phase 5 — Launch & handoff

Hand the client a running system, not just a scenario. The handoff defines whether this becomes a recurring relationship or a one-time job.

Steps
✓
Activate and monitor go-live Launch

Turn the scenario on with the client present. Run a live test together so they see it work end-to-end before you hand over the keys.

  • Use a real trigger — not a simulation — for the first go-live run
  • Watch the Make.com execution log in real time
  • Confirm the output appears in the destination (Airtable record, Google Drive file, email inbox)
  • Monitor the first 24–48 hours of real runs closely
✓
Deliver client documentation Handoff

Every engagement gets a handoff doc. This protects you from support requests and positions you as a professional. Keep it short — clients don't read essays.

  • What the automation does (one sentence per scenario)
  • What triggers it and what it produces
  • How to know if it's working (what to check)
  • What to do if something looks wrong (and who to contact)
  • What's explicitly out of scope
Deliverable
  • 1–2 page handoff doc (Notion, Google Doc, or PDF)
  • Loom walkthrough video (optional but high value)
✓
Set up monitoring and alerts Ops

You should know when something breaks before the client does. Set up a basic monitoring layer as part of every engagement.

  • Make.com email alerts on scenario errors (route to your email)
  • Weekly execution report for scenarios that run frequently
  • Airtable view or dashboard showing automation outputs (if applicable)
  • Agreed SLA for response time if something breaks
✓
Book the 30-day check-in and upsell Retention

Book a 30-day check-in call before you leave the handoff. This is the highest-leverage move in the entire engagement — it's where repeat work and referrals come from.

  • Is the automation running as expected?
  • What's changed in the workflow since launch?
  • What's the next highest-friction area you haven't touched yet?
Deliverable
  • Check-in call booked (Calendly link sent at handoff)
  • Phase 2 proposal ready if expansion is likely
The check-in call isn't support — it's a scoped conversation. If they want more work done, that's a new engagement.
