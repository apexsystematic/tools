Phase 01
Pre-Build Scoping
Before writing a single module, you need a locked definition of what you're building, for whom, and what success looks like. Ambiguity here costs you rebuild time later.
Requirements Capture
✓
Define the trigger event precisely
What event starts this automation? Form submission, email received, new Airtable row, webhook, scheduled time? Define the exact condition — not "when a client fills in a form" but "when a Tally form with ID X submits to webhook Y."
✓
Define the end state
What has the world looked like when this automation completes successfully? A row written to Airtable, an email sent, a Slack message posted. Measurable, not vague.
✓
Map every data input and output
List every field that enters the scenario (from the trigger) and every field that must be written or sent by the end. Mismatched field assumptions are the #1 cause of mid-build rework.
✓
Identify all external systems involved
List every app, API, or service the automation will touch. For each: confirm you have access, check API limits, and confirm auth method (OAuth, API key, webhook secret).
✓
Establish volume expectations
How many times per day/week will this trigger? High-volume scenarios have different design constraints (operations limits, rate limits, module costs on Make.com).
Scope Constraints
✓
Define what is explicitly out of scope
Write down what this automation does NOT do. Prevents scope creep and provides a reference for client conversations when new requests arrive mid-build.
✓
Identify edge cases upfront
What happens if a required field is empty? What if the upstream API is down? What if a duplicate is submitted? List likely edge cases now — you don't have to handle them all, but you need to consciously decide which ones to cover.
✓
Confirm deliverable format
Are you handing over a Make.com Blueprint JSON? A scenario they can duplicate? A shared team account? Confirm this before you start — it affects how you structure the build.
⚙
Make.com: Check the client's plan tier before starting. Free plan = no JS execution module. Core plan = limited operations/month. This changes your design options. Never assume they're on Growth.
△
DRI Check: If you haven't identified who owns each connected system (and can provide API keys/OAuth), do not start building. Missing access mid-build is a major time sink.

Phase 02
Architecture & Design Decisions
Design decisions made before building are cheap. Made during the build, they're expensive. Made after the build, they're painful.
Scenario Structure
✓
Choose single scenario vs. multi-scenario architecture
One long scenario or several chained ones? Single scenarios are simpler to debug but harder to maintain when complex. Split by logical domain (trigger → process → notify) when a scenario exceeds ~15 modules or contains multiple independent branches.
✓
Decide on routing logic upfront
Will you need conditional paths (Router module)? Map the branches before building. A Router with 4+ branches should prompt you to consider whether two separate scenarios would be cleaner.
✓
Plan your data structure before any module is added
Sketch the payload on paper or in a doc: what fields come in, how they transform, what goes out. Know your variable names before you touch the canvas. Naming inconsistencies compound fast.
✓
Choose AI model intentionally
Not every AI step needs Sonnet. Match model to task: Haiku for high-frequency structured extraction, Sonnet for judgment/synthesis, Opus only when output quality directly drives revenue (proposals, client-facing copy).
Design Principles
Idempotency — build to handle duplicate triggers safely ▼
Your automation should produce the same result whether it runs once or three times with the same input. Practically: before creating a record, check if it already exists. Before sending an email, check a "notified" flag. Use a unique ID from the trigger payload as your deduplication key.
⚙
Make.com: Use a Search Records module before any Create module. If a match is found, route to Update instead. Add a processed_at timestamp field to Airtable records as a simple idempotency marker.
Statelessness — don't rely on previous execution state ▼
Each scenario execution should be self-contained. Don't assume a previous run left data somewhere accessible. If you need to carry state between runs, store it explicitly (Airtable row, Data Store, Google Sheet cell). Make.com's Data Store is useful here for lightweight persistence.
Least privilege — only touch what you need ▼
When setting up OAuth connections or API keys, request only the permissions the scenario actually needs. Don't connect a Gmail integration with full account access if you only need to send emails. This limits blast radius if credentials are compromised.
△
This applies especially to Airtable: connect only the base/table the scenario needs, not your entire workspace.
Observable — make the automation self-documenting in its outputs ▼
Every execution should leave a trace. Write a last_run timestamp, a status field, and an error_log field to whatever data store you're using. This is how you debug production issues without needing to watch execution logs live.
✦
Design rule: If you can't explain what the scenario does in two sentences, it's trying to do too much. Split it.

Phase 03
Build Sequence & Methodology
How you build matters as much as what you build. A disciplined sequence catches problems early when they're cheap to fix.
The Happy Path First Rule
✓
Build the trigger + first module only, then test
Don't build the whole scenario before testing. Add the trigger, confirm it fires and returns the expected payload structure, then proceed. A bad assumption about field names at this stage will corrupt every downstream module.
✓
Build the full happy path end-to-end before adding any error handling
Get the core flow working with clean, valid input first. Don't add error branches, fallbacks, or edge case handling until the primary path is validated. Mixing the two makes debugging exponentially harder.
✓
Design and test AI prompts before wiring them in
Never write a prompt directly in a Make.com Claude module without testing it first. Test in the Anthropic Workbench or a standalone tool with real sample data. Only wire it in once output is consistently correct.
✓
Run a full end-to-end test with real data before adding error handling
Trigger the scenario with the most realistic test input you have. Verify every output — check the Airtable row was actually written, the email actually landed, the webhook actually received. Don't assume.
Build Hygiene
✓
Name every module descriptively
Default names like "HTTP 3" or "Router 1" are useless. Name them: "Check for existing contact in Airtable" or "Send intake confirmation email via Gmail." Takes 10 seconds, saves 10 minutes of debugging later.
✓
Use consistent variable naming conventions
Decide on a naming convention before you start: snake_case, camelCase, or prefixed (client_name, client_email). Stick to it. Mixed conventions in mapped fields create confusion during handoff and debugging.
✓
Add scenario notes for non-obvious logic
If a module does something that won't be obvious to someone reading the scenario cold, add a note. Especially: why a particular filter condition was chosen, or why a specific field is being transformed a certain way.
✓
Version control: export Blueprint JSON after each major milestone
Make.com doesn't have Git. Export a Blueprint JSON snapshot after: (1) happy path working, (2) error handling added, (3) final QA passed. Keep these in a project folder. If something breaks, you can roll back.
⚙
Make.com — Free plan workaround: No JavaScript module means no inline array manipulation. Handle array data from webhook payloads via direct field mapping (e.g. {{1.results[].score}}) or route into an iterator. Plan this before building, not after.

Phase 04
Error Handling Patterns
Errors are not exceptional — they are expected. Design your automation to degrade gracefully, alert appropriately, and never silently fail.
Error Classification
Error Type Example Correct Response
Hard Fail Missing required field, auth rejected Stop execution, alert, log
Soft Fail API timeout, rate limit hit Retry with backoff, then alert if persistent
Recoverable Duplicate detected, record not found Route to alternate path (Router), log, continue
Warning Optional field empty, partial data Continue, log warning, notify if threshold exceeded
Implementation Checklist
✓
Add error handlers to every API/HTTP module
Right-click any module in Make.com → Add error handler. At minimum, add a Rollback or Resume directive. For critical modules (payment, write operations), always use Break + alert.
✓
Never let the scenario silently complete with missing data
Use a Filter module after any data lookup to assert required fields are present before proceeding. If the filter fails, the execution stops — better than corrupted downstream records.
✓
Log every error to a dedicated Airtable error table
Create an "Error Log" table: scenario name, timestamp, error type, payload snapshot, module name. This is how you debug production issues without being present when they occur.
✓
Set up an alert for hard failures
On error routes: send a Gmail notification to yourself (and optionally the client) with: scenario name, time, error detail, and what data was being processed. You need to know when production breaks before the client does.
✓
Configure retry logic on transient failures
For HTTP requests, API calls, and AI module calls: enable retries in Make.com module settings. Set max retries to 3 with increasing intervals. Don't retry indefinitely — set a maximum and route to the error handler after.
✓
Handle incomplete executions
In Make.com scenario settings → enable "Allow storing incomplete executions." This lets you manually resolve and resume failed runs from the Incomplete Executions tab rather than losing the data entirely.
⚙
Make.com Error Handler pattern: On any critical module, add a route from the error handler that: (1) writes to your Airtable Error Log, (2) sends a Gmail alert to yourself, (3) uses the Break directive to stop the execution cleanly. Never use Ignore in production scenarios.
✕
Never use "Ignore" error directive in production. It silently swallows the error and marks the execution as successful. You will not know data was lost until the client tells you.

Phase 05
Security, Auth & Data Privacy
Your clients are lawyers and financial advisors. Their data is regulated. Getting this wrong isn't a product problem — it's a legal one. Build defensively from the start.
Authentication
✓
Use OAuth over API keys wherever available
OAuth tokens are scoped, revocable, and don't require you to handle raw credentials. If a service offers both, always choose OAuth. Reserve API keys for services that provide no alternative.
✓
All credentials belong to the client — not you
Every OAuth connection and API key must be created on the client's own accounts. You configure it, but they own it. If you hold their credentials and your account is compromised, their data is exposed. This is also your commercial position: client owns everything.
✓
Store API keys in Make.com's encrypted connections only
Never hardcode API keys into module fields, notes, or webhook URLs. Always store them in a named Make.com connection where they are encrypted at rest. If you see a key in a text field, it's wrong.
✓
Validate webhook signatures where supported
If your trigger is an inbound webhook, verify it's coming from the expected source. Most platforms (Stripe, Tally, etc.) sign their webhooks. Add a validation step at the top of your scenario before processing any payload.
Data Handling & Privacy
✓
Identify all PII in the data flow
Before building, list every field that could be considered personally identifiable: name, email, phone, DOB, NI number, company name, case reference. Know exactly where each field is written, stored, and sent.
✓
Don't pass PII to AI models unnecessarily
If you're using Claude for document classification or summarisation, strip or pseudonymise PII before sending to the API where possible. Pass reference IDs, not personal details. Only include what the model actually needs to perform the task.
✓
Confirm data residency requirements
For UK/EU clients, confirm whether data can transit through US-based services. Make.com is US-based. Anthropic's API is US-based. Some regulated industries (law, financial services) have restrictions on where client data can be processed.
✓
Minimise data retention
Only store what is required for the automation to function. Don't accumulate client data in your own Make.com Data Stores or Airtable bases by default. If storing is necessary, define a retention period and communicate it.
✓
Document what data transits through Make.com
For each scenario, create a simple data flow note: what enters, what's processed, what's stored, what's sent externally. This becomes part of the client handoff documentation and covers you if questions arise later.
△
Notion integration: The most common setup failure is forgetting to share the Notion database explicitly with the Make.com integration inside Notion. The connection will authenticate fine but reads/writes will silently fail. Always test with a real operation, not just a connection check.
✕
For law firm and IFA clients: Before handling any client matter data through your automation, confirm in writing what data will transit through which services. Do not assume they've considered this. Flag it proactively — it protects both of you.

Phase 06
Testing & QA
Testing is not checking if it works once. It's proving it works correctly across all expected inputs, and fails safely on unexpected ones.
Test Matrix — Run All Before Sign-Off
✓
Happy path: clean, complete input data
Trigger with a complete, valid payload — every field populated, all values in expected format. Verify every output exactly: correct data in the right fields, emails sent to the right address, records created not updated.
✓
Missing optional fields
Remove optional fields from the trigger payload one at a time. Does the scenario handle their absence gracefully, or does it error? Fix any crashes caused by missing optional data.
✓
Missing required fields
Remove a required field. The scenario should fail fast and cleanly — hitting the error handler, logging, and alerting — not silently proceed with null data being written downstream.
✓
Duplicate trigger
Submit the exact same input twice. Does the second execution create a duplicate record, or does your idempotency logic catch it and route to an update/skip path?
✓
Edge case inputs
Test with: very long strings, special characters (apostrophes, ampersands), international characters, empty strings vs. null vs. whitespace-only. These are where integrations break silently.
✓
Simulated API failure
Temporarily break a downstream connection (wrong API key, invalid endpoint) and trigger the scenario. Confirm the error handler fires, the error is logged, and the alert email is sent. This is the only way to verify your error handling actually works.
✓
High-volume / rapid succession
For scenarios that may receive bursts of triggers: fire 5–10 in quick succession. Check for race conditions, duplicate processing, or rate limit errors. Review execution logs for any that silently failed.
Sign-Off Criteria
✓
All test cases in the matrix passed
Every scenario in the test matrix has been run and verified. No untested paths remain.
✓
Blueprint JSON exported and archived
Final scenario exported as Blueprint JSON, named with version and date (e.g. law-intake-v1.2-2026-04.json), stored in project folder in Google Drive.
✓
Error log table is empty
After all test runs, review the Airtable Error Log. It should be empty (or contain only intentional test errors you've cleared). Any unresolved errors = not ready for handoff.
✓
Scenario is off / inactive before handoff
Turn the scenario off before handing over. The client should be the one to activate it on their account for the first time. This prevents test executions from contaminating their live environment during the import process.
✦
30-day check-in: Book a 30-minute call at handoff, 30 days out. Position it as part of the delivery, not optional support. This is your best upsell window — by then they know what else they want automated.

DirectiveBehaviourUse When
Break Stops the execution. Marks it as Incomplete (can be resumed manually). Critical failure — data must not be lost. Allows manual intervention and retry.
Rollback Undoes all operations in the current execution bundle. When partial writes are worse than no write. Use when atomicity matters.

Make.com free plan: no JavaScript execution module ▼
Array manipulation from webhook payloads must use direct field mapping or an Iterator module. You cannot use Set Multiple Variables with JS expressions. Design around this before you start — retrofitting is painful.
Notion: database must be explicitly shared with Make integration ▼
The OAuth connection will succeed, but reads and writes will silently fail unless you open the Notion database → Share → explicitly add the Make.com integration. This is the #1 setup failure. Always test with a real write operation, not just a connection check.
TikTok Content Posting API requires pre-approved developer access ▼
You cannot post to TikTok programmatically without applying for and receiving approval from TikTok's developer program. Always flag this as a conditional deliverable in proposals and never commit to a delivery date until approval is confirmed.
Make.com: "Ignore" error directive masks failures ▼
If you add an error handler with "Ignore," the execution shows as successful in logs even if a module failed. You will not know data was lost or not processed. Never use Ignore in production. Use Break + log + alert instead.
Airtable: API rate limit is 5 requests/second per base ▼
For high-volume scenarios with multiple Airtable operations per execution, you can hit the rate limit quickly. Add a Sleep module between writes, or batch operations where possible. Use the Airtable API directly via HTTP module for bulk operations — the native Make.com module doesn't support batch writes.
Anthropic API: long prompts + large context = slow + expensive ▼
Don't pass entire documents to Claude unless necessary. Extract only the relevant sections before sending. Set a reasonable max_tokens limit — open-ended token limits on high-volume scenarios will burn budget fast. For structured extraction tasks, use a strict JSON output format instruction to keep responses short.
Webhook payload: field names are case-sensitive ▼
If your form sends clientName and you map ClientName, the field will be empty and downstream modules will fail silently. Always inspect the raw webhook payload in Make.com's execution history before mapping fields. Never map from memory.
